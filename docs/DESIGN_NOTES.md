# DESIGN_NOTES: ホームページ刷新 (DX特化への転換) 決定台帳

追記専用。決定を覆すときは旧行末尾に「→ 撤回」を付記し新行を追記する。

## 背景・前提

- 課題: 現ホームページは「誰に何を訴えるか」が不明瞭で Web 集客実績ゼロ。IT 企業向け高難度受託の顔 (Rust/Go/IoT) と営業資料の顔 (中小企業の現場DX) が乖離している
- 議論の経緯: 3 ロール (買い手代弁者 / BtoB マーケ実務家 / Devil's Advocate) の debate をラウンド 2 で収束済み (2026-07-19)

## 決定台帳

### 戦略 (debate で確定、ユーザ承認済み 2026-07-19)

- [決定] ターゲットは非IT中小企業の経営者 (ホテル・食品・運送など、社員10〜100名規模) — 営業資料 dx.html と同一ターゲットで顔を一致させるため
- [決定] HP の役割は「紹介・営業後の検証装置」(紹介・営業接触後の社名検索の受け皿)。直接リード獲得は主目的にしない — 非IT中小企業経営者は Web 検索で開発会社を探さず、紹介後に身元調査として HP を見るため
- [決定] トップの訴求は業種ではなく課題の語彙で絞る (「紙・Excel・属人運用の現場業務をなくす」系) + 業種不問を明記 — 「DX」の抽象連呼は補助金業者と識別不能になり、業種絞りは多業種の紹介訪問者を弾くため
- [決定] キーメッセージは営業資料 dx.html の「作って終わりにしない。現場に根付くDXを、一緒に。」と一致させる — 営業で検証済みのメッセージを Web 化する方針
- [棄却] 「いろんな業種のDXに強い」という横断抽象訴求をトップコピーにする — 何も頭に残らず自分事化しないため (3 ロール一致)
- [棄却] 実績のある業種 (ホテル/食品/運送) への業種絞り込み訴求 — 紹介経由で来る多業種の経営者を入口で弾くため。絞り込みは業種でなく課題語彙で行う
- [決定] 事例 3 件 (リゾートホテル業務管理 / アレルゲンチェックシステム / 運送会社書類提出DX) は個別ページ化し「課題 → 現場がどう変わったか」の物語で書く。業種ラベルは付けるが入口は課題訴求
- [決定] 事例は匿名で書く (実名許諾は取れない)。信用力低下は数値と代表の顔で補強する — ユーザ確認済み (2026-07-19) → 一部撤回 (代表の顔での補強は撤回、詳細は「監査反映」節末尾を参照)
- [決定] 料金の目安幅を掲載する — 相場不安が最大の離脱要因。目安幅の足切り機能が値切りリスクに勝る (買い手・マーケ双方が譲歩不可とした唯一の共通項)
- [決定] 主 CTA は「30分無料相談・準備不要」。フォーム単独にしない — 要件を文章で書ける経営者はいないため
- [決定] 支援の進め方 (段階的定着プロセス) を dx.html から転載する
- [決定] KPI は 30 分無料相談の予約数 (月 1〜2 件で成功)。流入総数は KPI にしない — 検証装置という役割定義と整合させるため
- [決定] 事例記事のロングテール SEO は「長期の副産物」と位置付ける (6〜12 ヶ月で月数百 UV・問い合わせ 0〜1 件が現実的期待値) — 集客装置としての期待はしない
- [決定] 既存のエンジニア向けコンテンツ (高難度受託実績・ギークの楽園ビジョン・技術スタック等) は削除せず「エンジニア向けページ」として温存する — エンジニア受けが良く採用の受け皿になるため。ユーザ確認済み (2026-07-19)
- [棄却] エンジニア向けコンテンツの完全撤去 (GitHub・技術発信への外部化のみで担保) — ユーザ判断で HP 内温存に決定

### 構造 (ユーザ承認済み 2026-07-19)

- [決定] 共存構造はトップ「/」を経営者向け DX に全面刷新し、エンジニア向けコンテンツ (技術実績・技術スタック・企業理念・採用) は /engineers 配下へ集約する。ヘッダーは経営者向けナビ、端に小さく「エンジニアの方へ」リンク — 検証装置として経営者の第一印象を最優先するため
- [棄却] トップを経営者/エンジニアの 2 ドア選択ページにする — 経営者の第一印象が 1 クリック遅れるため
- [棄却] サブドメイン分離 (engineers.techlead-it.com 等) — Worker 構成・運用が 2 倍になるため
- [決定] 既存ページの帰属: /about は共用維持 (実在性確認に必須)、/news は共用維持、/slides は共用維持 (研修・伴走支援の証拠として経営者にも見せる)、/introduction (企業理念・ギークの楽園) はエンジニア向け配下へ移設
- [決定] 匿名事例の数値は営業資料 dx.html 記載の実数値を使う — ホテル: 確認・引き継ぎ時間 約50%削減 / アレルゲン: 確認作業時間 約70%削減 / 運送: 書類作成時間 約80%削減 (数値ソース確認済み 2026-07-19)
- [決定] 相談導線は既存問い合わせフォームの拡張のみ (「30分無料相談希望」区分の追加)。予約ツール・電話掲載はしない — ユーザ判断
- [棄却] Google カレンダー等の日程直接予約ツール導入 — ユーザ判断で不採用

### 情報設計 (ユーザ承認済み 2026-07-19)

- [決定] /engineers はハブ + 理念サブページ構成。/engineers (技術実績・技術スタック・採用) + /engineers/philosophy (旧 /introduction の理念コンテンツをほぼそのまま移設) — 既存コンテンツの改修量を抑えるため。ユーザ決定
- [決定] 料金目安はトップのセクションとして掲載し、独立ページは作らない — 身元調査 (紹介後の社名検索) で 1 ページ完結で金額の桁まで見えることを優先。ユーザ決定
- [決定] 匿名事例の視覚素材は before/after の業務フロー図解のみ。実物スクリーンショット・デモ URL は掲載しない — 匿名性維持と許諾不要を両立するため。ユーザ決定
- [決定] 匿名表記ルールは「業種 + 規模感」(例: リゾートホテル (数十室規模) / 食品関連企業 / 運送会社 (車両数十台規模)) — 自分事化に必要な最小情報に絞る。※Claude 提案 (規模感の具体表記は事実と整合する範囲でライティング時に確定)
- [決定] サイトマップ: 経営者向け = `/` (全面刷新) + `/cases` (事例一覧・新設) + `/cases/:id` (事例詳細 3 件・新設) + `/contact` (30分無料相談区分を追加改修)。共用 = `/about` `/news` `/news/:id` `/slides` `/slides/:categoryId`。エンジニア向け = `/engineers` (新設ハブ) + `/engineers/philosophy` (移設)
- [決定] 旧 `/introduction` は `/engineers/philosophy` へ SPA ルートでリダイレクトする — 既存リンク・被リンクの断絶防止。※Claude 提案
- [決定] ヘッダーナビは経営者向け (支援内容 / 事例 / 料金 / 会社概要 / お知らせ / 無料相談 CTA) + 右端に控えめな「エンジニアの方へ」リンク
- [決定] 事例数値のラベル確定 (dx.html 突合済み 2026-07-19): ホテル = 確認・引き継ぎ時間 約50%削減・取りこぼしも減少 / アレルゲン = 確認作業時間 約70%削減・見落としリスクも大幅低減 / 運送 = 書類作成時間 約80%削減・提出漏れ・期限切れをゼロに

### 監査反映 (2026-07-19 utility-doc-audit の指摘を受けた決定。※Claude 提案、ユーザレビュー待ち)

- [前提追記] 集客リソースはコンテンツ執筆のみ (広告予算・専任営業なし) — debate の前提事実を台帳に正本化 (監査指摘: 台帳に根拠なき制約記述)
- [決定] 相談区分は新フィールドを追加せず、既存 subject 選択肢の先頭に「30分無料相談」を追加して初期選択にする — スキーマ・メールテンプレート変更ゼロで「相談区分の追加」決定を満たす最小実装 (subject はメール本文に表示済みのため KPI 集計にもそのまま使える)
- [決定] KPI 計測は受信メールの件名区分「30分無料相談」の月次目視集計とする — 追加実装なし
- [決定] トップの既存セクションコンポーネントの行き先: HeroSection=経営者向けに改修 / NewsSection=トップから外し削除 (使用箇所が Home のみのため) / ServicesSection=「支援できること」に改修 / StrengthsSection=「信用補強」に改修 / CtaSection=相談 CTA に改修 / ProjectsSection・ProcessSection・TechStackSection=/engineers へ移設
- [決定] トップに NewsSection (お知らせ抜粋) は置かない — 検証装置の役割に不要。/news はヘッダー・フッターから到達
- [決定] dx.html 転用コンテンツのデータ配置: 悩み提示 = painPoints.ts (新設) / 支援できること = services.ts (経営者語彙に改修) / 段階的定着プロセス = dxProcess.ts (新設) / 料金目安 = pricing.ts (新設)
- [決定] トップの「支援の進め方」(dxProcess.ts) と /engineers の「開発の進め方」(既存 processSteps.ts) は別データとして分離する — 経営者語彙とエンジニア語彙は両立しないため
- [決定] 信用補強セクションのデータは strengths.ts を経営者語彙 (伴走姿勢・誠実さ) に改修して流用し、代表プロフィール (顔写真・氏名・経歴) は company.ts を拡張して保持する → 撤回 (下記ユーザ判断により)
- [決定] CaseStudy 型は industry (業種) / scale (規模感) / supportScope (支援範囲) を個別フィールドで持つ (業種+規模感の結合文字列 industryLabel は不採用) — 事例詳細の概要表と型の 1:1 対応のため
- [決定] /slides への導線は共通フッターに「研修・資料」として残す (経営者向けヘッダーには置かない)。zenn 技術ブログリンクは経営者向けヘッダーから外し /engineers とフッターへ移す
- [決定] ページ別 meta (title) は useEffect + document.title の軽量実装とする (meta ライブラリは導入しない)。事例 OGP 画像生成はオプション扱いのまま、既存 OGP 生成の流用方法は実装時判断
- [決定] 表記の正: CTA 文言は「30分無料相談」(スペースなし)、数値は「約50%削減」のように「削減」まで含めた文字列を正式値とする、アレルゲン事例の業種ラベルは「食品を扱う事業者」(旧記載「食品関連企業」を更新)
